# 实验一 EDA 可视化：图注、正文解读与结论

> 本文档按 Nature Commun. 图注规范撰写。每节包含：
> - **图注**（插入论文/报告时放置于图片下方）
> - **正文解读**（在报告相应段落引用该图时的叙述文字）

## 图 1　`merge_vs_nonmerge.png`

### 图注

**图 1 | Merge Prediction 目标标签的分布与跨仓库差异。**
**a** 合并与未合并 PR 的数量柱状图（Merged = 1,150，Non-merged = 556，N = 1,706）。
**b** 类别占比甜甜圈图，整体合并率为 67.4%，正负样本比约 2:1，存在轻度类别不平衡。
**c** 各仓库合并率横向条形图（按合并率升序排列），虚线表示全局均值（67.4%）。
仓库间合并率差异显著：django/django 最低（47.8%），home-assistant/core 最高（90.2%），
跨度超过 42 个百分点。*n* 标注于每条柱末。

### 正文解读

本数据集共收录 1,706 条已关闭 PR，其中 1,150 条（67.4%）被合并，556 条（32.6%）以未合并状态关闭（图 1a、b）。整体正负样本比约为 2:1，属于轻度类别不平衡——这与代码审查领域的普遍规律相符，大量贡献最终被接受，但仍有相当比例因质量、方向或流程问题被拒绝。

更值得注意的是各仓库间的差异（图 1c）。home-assistant/core 和 apache/airflow 的合并率分别高达 90.2% 和 79.5%，而 django/django 和 huggingface/transformers 分别仅为 47.8% 和 55.5%。这一差异反映了不同开源社区迥异的 PR 准入文化：django 以严格的代码审查和高标准著称，贡献者需反复修改；home-assistant 则更多依赖自动化测试和 bot reviewer，符合质量标准的 PR 通常可快速合并。

### 结论

> **类别不平衡问题（对应指导书思考题 5）**：数据集整体正负样本比约 2:1（67.4% vs. 32.6%），实验二/三构建 Merge Prediction 分类器时需采用分层采样（stratified split）或在损失函数中引入类别权重以缓解偏差。
>
> **分布偏移警告**：各仓库合并率跨度超过 40 个百分点，跨仓库训练/测试时模型性能将受数据来源分布的显著影响；建议按仓库分层划分训练集/测试集，或将仓库 ID 作为协变量加入模型。

## 图 2　`review_comment_dist.png`

### 图注

**图 2 | Inline Review Comment 的零膨胀分布与社区审查密度差异。**
**a** 全数据集 inline review comment 数量直方图（截尾至 p98 = 25 条），竖虚线为中位数（0 条）。
标注框汇报关键分位数：73.9% 的 PR 无任何 inline 评论；p90 = 5 条，p99 = 24 条，最大值 202 条（N = 1,706）。
**b** 各仓库"有 ≥1 条 inline 评论"的 PR 占比横向条形图；括号内 μ 为仓库内均值。

### 正文解读

Inline review comment（钉在具体代码行上的审查意见）是实验三"Review Comment Generation"任务的唯一训练数据来源，其分布直接决定可用训练样本的规模。如图 2a 所示，该分布呈现极强的零膨胀特征：73.9% 的 PR 在整个审查过程中没有产生任何 inline 评论，中位数为 0。仅在尾部存在少量多评论 PR（p90 = 5 条，p99 = 24 条，最大值高达 202 条），整体均值被这些异常值拉高至 1.87 条。

在社区层面（图 2b），django/django 的 inline 评论密度最高（30.3% 的 PR 含有评论，均值 2.45 条），与其以细致代码审查著称的社区文化相符。pandas-dev/pandas 比例最低（24.1%），但从均值角度（1.12 条）与 home-assistant/core 相当。

### 结论

> **实验三数据量预估**：仅约 26% 的 PR（约 444 条）产生可用的 `(diff_hunk → 审查意见)` 配对，是生成任务的实际可训练样本上界。实验三划分数据集时必须先过滤 `num_review_comments > 0`，否则将引入大量无标签噪声。
>
> **社区差异**：django 审查密度最高（30.3%）；如需构建审查意见质量更高的子集，可优先选取 django 和 huggingface/transformers 的 PR。

## 图 3　`label_dist.png`

### 图注

**图 3 | PR 标签（Label）频次分布（Top-15）。**
横向条形图展示 PR 标签的出现频次（N = 1,706 条 PR，共 419 种不同标签，总计使用 3,530 次）。
条形颜色深度与频次正相关（蓝色梯度，越深越频繁）。末端数值为出现次数。

### 正文解读

PR 标签系统是各仓库分类与流程管理的重要元数据。如图 3 所示，使用频次最高的标签为 `cla-signed`（313 次）——这是 apache/airflow 等仓库要求贡献者签署贡献者许可协议（CLA）的自动化标记，反映了大型开源项目的法务合规流程。`small-pr`（270 次）、`code-quality`（195 次）和 `has-tests`（117 次）则是 home-assistant/core 等仓库对 PR 规模和质量的程序化分类标签。前 15 个标签合计覆盖约 47% 的总标签使用量，表明标签分布同样呈现幂律形态——少数高频标签主导了整体分类体系。

### 结论

> 标签作为元数据特征对 Merge Prediction 有一定价值（实验二可将标签数量 `num_labels` 纳入统计特征），但原始标签字符串因仓库而异，不宜跨仓库直接使用；若用于建模，建议先归一化（如只保留跨仓库通用语义的标签类别）。

## 图 4　`reviewer_dist.png`

### 图注

**图 4 | Reviewer 数量分布及其与合并结果的关系。**
**a** 全数据集每条 PR 参与不同 reviewer 数量的柱状分布（N = 1,706，均值 = 1.30）；
柱顶标注绝对计数。
**b** 按 reviewer 数量（0/1/2/3/4+）分组后的合并率柱状图；
虚线为全局合并率（67.4%）；括号内为各组样本量。

### 正文解读

如图 4a 所示，reviewer 数量集中在 0–2 之间：无 reviewer 的 PR 占 30.6%（522 条），1 名 reviewer 占 28.4%，2 名占 27.6%，3 名及以上合计仅占 13.4%。均值为 1.30，中位数为 1，体现了大多数 PR 由单一维护者裁定的典型开源工作流。

图 4b 揭示了 reviewer 数量与合并率之间显著的正相关关系：无 reviewer 的 PR 合并率仅为 27.0%，而拥有 2 名及以上 reviewer 的 PR 合并率跃升至 92.6% 和 89.5%。这一趋势的合理解释是：被多名维护者主动 review 的 PR，通常是优先级更高、质量更过关的变更，在流程上也更接近主干维护者的预期。

### 结论

> `num_reviewers` 是 Merge Prediction 中信息量最高的统计特征之一（见合并率从 27% 到 93% 的跨度），实验二构建特征工程时应优先纳入。但需注意因果方向：reviewer 数量多并非导致合并的直接原因，而是"PR 被重点关注"这一潜在变量的代理指标。

## 图 5　`pr_size_dist.png`

### 图注

**图 5 | PR 改动规模的重尾分布及跨仓库差异。**
**a** PR 大小（additions + deletions）的对数直方图（N = 1,706）；三条竖线分别标注中位数
p50 = 32 行（红实线）、p90 = 339 行（金虚线）、p99 = 4,809 行（灰点线）；分布横跨约 6 个数量级，
最大值达 971,560 行。
**b** 各仓库 PR 大小的小提琴图（横轴为 log₁₀ 改动行数，按中位数升序排列），
黑线为组内中位数。pandas-dev/pandas 整体最大。

### 正文解读

PR 改动规模是最典型的重尾（heavy-tailed）特征。如图 5a 所示，绝大多数 PR 极小——中位数仅 32 行，但分布尾部极长：p90 为 339 行，p99 达 4,809 行，最大值高达约 97 万行（通常是自动生成文件、依赖锁定文件或大规模数据迁移）。分布横跨约 6 个数量级，因此必须使用对数坐标；线性直方图会被极端值完全压缩、失去可读性。

跨仓库来看（图 5b），pandas-dev/pandas 的 PR 规模中位数显著高于其他仓库，因其 PR 常附带大量测试用例与文档改动；django/django 则以小而精的变更为主，符合其严格审查、鼓励拆分小 PR 的文化。

### 结论

> **特征工程注意**：PR 大小类特征（`additions`、`deletions`、`changed_files`）在建模前必须做对数变换（log1p）或分位数归一化，否则少数极端 PR 会主导 SVM 的核计算与距离度量。
>
> **实验二/三预处理**：建议对超大 PR（如 > p99）设置截断或单独标记，避免自动生成的巨型 diff 污染 AST/CFG 特征提取与 CodeBERT 的 token 截断。

## 图 6　`ai_vs_human.png`

### 图注

**图 6 | AI 参与度：信号构成及其与合并结果的关联。**
**a** `is_ai_code` 命中信号的类型构成饼图（蓝色为强信号：commit/body 中的 Co-authored-by、AI bot 作者；
灰色为弱信号：AI 相关 label、文本提及）；强信号占绝大多数。
**b** AI 生成代码 vs 人类代码 PR 的合并率对比（AI code n = 294，Human code n = 1,412），
虚线为全局合并率。
**c** 有/无 AI reviewer 的 PR 合并率对比（Has AI reviewer n = 323）。
各柱标注合并率与样本量。

### 正文解读

AI 标签是实验五（AI 生成代码审查）的数据基础。判定采用启发式信号，其可靠性取决于信号强度。如图 6a 所示，`is_ai_code` 的判定绝大多数来自强信号——commit message 中的 `Co-authored-by: <AI 工具>`（376 次）占主导，弱信号（label、文本提及）仅占少数，说明 AI 代码标签整体具备较高可信度。

在合并结果上，AI 生成代码与人类代码的合并率接近（71.8% vs. 66.5%，图 6b），差异有限，表明 AI 辅助生成的代码在被接受度上并未明显劣于人类代码。相比之下，AI reviewer 的存在与合并结果高度相关（图 6c）：有 AI reviewer 的 PR 合并率高达 87.9%，远高于无 AI reviewer 的 62.6%。但需谨慎解读因果——这很可能是选择效应：home-assistant/core 等高合并率仓库恰好大规模部署了 Copilot PR reviewer（该仓库贡献了 323 条 AI reviewer 样本中的 301 条）。

### 结论

> **实验五样本充足性**：`is_ai_code = True` 达 294 条、`has_ai_reviewer = True` 达 323 条，满足实验五分析需求。若需高精度子集，应通过 `ai_code_signals` 列筛选强信号行（Co-authored-by / author_bot）。
>
> **混杂变量警告**：AI reviewer 与合并率的强关联受仓库混杂（confounding）影响，实验五分析 AI reviewer 效应时须控制仓库变量，不能直接归因于 AI 审查本身。

## 图 7　`cross_repo.png`

### 图注

**图 7 | 跨仓库多维对比：不同社区的代码审查文化差异。**
三联横向条形图（各仓库按合并率升序排列，配色跨图一致）。
**a** 各仓库合并率，虚线为全局均值（67.4%）。
**b** 各仓库每条 PR 平均 inline 评论数。
**c** 各仓库 PR 大小中位数（改动行数）。N = 1,706，五仓库各约 326–347 条 PR。

### 正文解读

图 7 从合并率、审查密度、改动规模三个维度横向对比五个仓库，直接呼应实验一拓展目标（分析不同开源社区代码审查流程的差异）。三个维度共同勾勒出鲜明的社区画像：

- **django/django**：合并率最低（47.8%）、审查评论密度最高（均值 2.45 条）、PR 规模最小。典型的"高门槛、精细审查、小步提交"文化。
- **home-assistant/core**：合并率最高（90.2%）、评论密度最低（1.18 条）。依赖自动化测试与 bot 审查，人工 inline 评论较少但通过率高。
- **pandas-dev/pandas**：PR 规模中位数最大，因 PR 普遍附带测试与文档。
- **huggingface/transformers**：合并率偏低（55.5%）但评论密度较高（3.19 条），体现活跃且严格的社区讨论。

### 结论

> 三个维度上的系统性社区差异证实：将五个仓库简单混合成单一训练集会掩盖社区间的分布差异。后续所有建模实验（二/三/四）都应将"仓库"作为分层或协变量因素处理，并在评估时报告分仓库指标，而非仅看整体平均。

## 图 8　`eda_overview.png`

### 图注

**图 8 | 实验一数据集综合概览（3×3 九宫格）。**
一图汇总数据集九个关键维度（N = 1,706 PR，5 仓库，2015–2026）：
**a** 合并结果占比；**b** inline 评论数分布（截尾 p98）；**c** PR 大小（对数）；
**d** reviewer 数量分布；**e** 正式 review 决策（APPROVED/NONE/COMMENTED/CHANGES_REQUESTED）；
**f** 作者身份（author_association）；**g** 各仓库 PR 数量；**h** AI 参与（is_ai_code / has_ai_reviewer）；
**i** PR 生命周期（open→close 耗时，对数，中位数 9.5 小时）。

### 正文解读

图 8 作为数据集的"体检报告"，在单张图中覆盖了指导书步骤五要求的全部统计维度，适合置于实验报告"数据集统计"章节开头作为总览。除前述维度外，本图补充了三个此前未单独展开的观察：

- **审查决策（e）**：962 条 PR 获正式 APPROVED，522 条无正式 review 决策（NONE，多为直接合并的小 PR），仅 80 条被明确要求修改（CHANGES_REQUESTED），说明显式拒绝在数据中较稀少。
- **作者身份（f）**：CONTRIBUTOR（736）与 MEMBER（582）占主导，外部无关联作者（NONE，326）占比不低，样本覆盖了从核心成员到外部贡献者的完整身份谱系。
- **PR 生命周期（i）**：从创建到关闭的耗时中位数仅 9.5 小时，但同样重尾（p90 = 180 小时 ≈ 7.5 天），反映"多数 PR 快速处理、少数长期悬置"的典型协作节奏。

### 结论

> 本概览图确认数据集在标签、规模、审查强度、AI 参与、时间跨度等维度上均具备足够的多样性与覆盖度，可作为后续 6 个实验统一、可靠的数据基础。多个维度（PR 大小、评论数、生命周期）共享重尾特征，提示后续建模需系统性地进行对数/分位数变换与异常值处理。
