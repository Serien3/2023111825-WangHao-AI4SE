### 1.3 实验背景

随着 GitHub、GitLab 等协同开发平台的广泛应用，代码审查（Code Review）已经成为现代软件开发过程中保证软件质量的重要环节。开发人员通常通过 Pull Request（PR）的形式提交代码修改，并由项目维护者或其他开发人员进行审查、讨论和修改建议，最终决定是否将代码合并到主分支。近年来，大语言模型（Large Language Models，LLMs）的快速发展进一步改变了代码审查过程。一方面，大量代码已经由 GitHub Copilot、ChatGPT、Claude Code、Cursor 等工具辅助生成；另一方面，越来越多的代码审查意见也开始由 AI 自动生成。因此，现代代码审查已经逐渐形成了“人类开发者 + AI 开发者 + AI Reviewer”共同参与的软件开发模式。为了研究不同代码审查方法的性能，本课程首先需要构建一个包含丰富信息的数据集，包括：

- Pull Request 基本信息；

- 提交代码；

- Code Review Comments；

- Review Decision；

- Merge 状态；

- 是否包含 AI 生成代码；

- 是否存在 AI Reviewer；

- 时间、标签等元数据。

该数据集将作为整个课程后续所有实验的数据基础。

#### 1.4.2 代码审查流程

代码审查主要包含以下几个阶段：

1. 提交代码修改；

2. Reviewer 阅读代码；

3. Reviewer 提交修改建议；

4. Developer 回复意见；

5. 修改代码；

6. 最终决定是否 Merge。

代码审查不仅能够发现软件缺陷，还能够提高代码可维护性、统一编码规范，并促进团队成员之间的知识共享。

```text

├── Issue Comments
├── Labels
```

这些对象共同构成了代码审查任务的重要数据来源。

#### 1.4.4 代码审查任务

本课程主要关注两个典型任务：

1. Merge Prediction

   判断一个 Pull Request 是否最终能够被项目接受。

### 2.4 实验原理

```text

↓
Merge Prediction
```

#### 2.4.2 代码中间表示

程序代码可以转换为不同的中间表示形式，从不同角度描述程序语义。

- 抽象语法树（Abstract Syntax Tree，AST）：描述程序语法结构；

- 控制流图（Control Flow Graph，CFG）：描述程序执行流程；

- Diff 信息：描述代码修改内容；

- Commit Message：描述开发者修改意图。

不同表示能够提供互补的信息，共同支持后续特征提取。

#### 2.4.3 特征工程

本实验主要构建以下几类特征：

- 代码结构特征（AST 节点数量、CFG 节点数量等）；

- 代码修改特征（新增代码行数、删除代码行数、修改文件数量等）；

- 文本特征（Title、Body、Commit Message 长度等）。

上述特征经过归一化处理后构成机器学习模型输入。

#### 2.4.4 传统机器学习模型

本实验主要采用以下两种经典模型：

1. 支持向量机（Support Vector Machine，SVM）

   SVM 通过寻找最优分类超平面对不同类别进行划分，适用于中小规模数据集。

2. 随机森林（Random Forest）

   随机森林由多个决策树组成，通过集成学习提高模型泛化能力，并能够分析各特征的重要程度。

### 3.3 实验背景

### 3.4 实验原理

#### 3.4.1 深度学习代码审查流程

深度学习代码审查通常包括以下几个步骤：

```text
代码输入
↓
代码表示学习
↓
深度神经网络编码
↓
任务微调
↓
Merge Prediction / Review Comment Generation
```

与传统机器学习不同，深度学习能够自动学习代码语义表示，无需人工设计大量特征。

#### 3.4.2 Code2Vec 模型

Code2Vec 是一种基于抽象语法树路径（AST Paths）的代码表示学习模型。该模型首先从程序抽象语法树中提取大量路径，再利用注意力机制学习不同路径的重要程度，最终生成整个代码片段的向量表示。Code2Vec 能够有效捕获程序的语法结构信息，适用于代码分类、代码克隆检测以及代码审查等任务。

#### 3.4.3 CodeBERT 模型

CodeBERT 是一种面向源代码与自然语言联合训练的预训练模型。模型利用 Transformer 结构，通过海量开源代码进行预训练，能够学习程序语义和自然语言描述之间的对应关系。由于 CodeBERT 已经学习了丰富的软件开发知识，因此能够较好地完成代码理解、代码搜索、代码摘要以及代码审查等任务。

#### 3.4.4 代码审查任务

本实验主要完成以下两个任务：

1. Merge Prediction

   根据 Pull Request 修改内容预测该 Pull Request 最终是否能够被项目接受并合并。

2. Review Comment Generation

   根据 Pull Request 修改内容自动生成合理的代码审查意见，为开发人员提供修改建议。

### 7.4 实验原理

#### 7.4.1 VSCode 插件架构

VSCode 插件通常包括以下几个组成部分：

```text
VSCode
↓
Extension API
↓
插件逻辑
↓
代码审查模型
↓
结果展示
```

插件通过 Extension API 获取编辑器中的代码信息，并调用后端模型完成智能代码审查。

#### 7.4.2 插件功能设计

本实验设计的 VSCode 插件主要包括以下功能：

- 获取当前编辑文件；

- 获取代码修改内容；

- 调用代码审查模型；

- 展示 Merge Prediction 结果；

- 展示自动生成的代码审查意见；

- 查看历史代码审查记录。

#### 7.4.3 模型集成方式

为了便于模型部署，本实验采用前后端分离方式：

- VSCode 插件负责用户交互；

- Python 后端负责调用机器学习、深度学习及大语言模型；

- 前后端通过 HTTP 接口进行通信。

这种架构能够方便模型更新，同时保证插件具有良好的扩展性。

#### 7.4.4 结果可视化

为了帮助开发者快速理解代码质量，本实验设计以下展示方式：

- Merge Prediction 结果；

- 代码审查意见列表；

- 风险等级提示；

- 高亮显示存在问题的代码位置；

- 模型推理时间及调用信息。
