# 实验六结果分析：改进针对大模型生成代码的代码审查

> 本文依据实验指导书第 6 章，对实验六全部落盘结果进行数据驱动分析。实验仅优化输入端的
> **软件工程上下文与 Prompt**，执行模型始终为 `deepseek-v4-flash`；生成任务另由
> `deepseek-v4-pro` 裁判。分析以逐样本预测为基础，图中区间为固定随机种子、2,000 次
> bootstrap 的 95% 置信区间。

## 1. 实验目标与证据设计

实验五发现了两个会误导结论的“指标失明”：AI 分类池的合并率为 74%，近似“全部判为 merge”
也能得到约 0.74 accuracy 和 0.85 merge-F1；开放式审查意见与单一参考文本的 BLEU/ROUGE
接近零，无法衡量意见是否正确、有用。实验六因此不以提高旧指标为唯一目标，而是回答三个更具体的
问题：

1. 深层软件工程上下文能否提高模型对 AI 代码的真实判别力和意见质量？
2. Self-Reflection 或多轮 Prompt 能否抑制“全部 merge”偏向？
3. 上下文和 Prompt 的收益是否足以抵偿额外时延，并能否迁移到人类代码？

分类主指标为 balanced accuracy、non-merge recall 和逐类 recall；生成主指标为裁判给出的
relevance、actionable、correct、hallucination 四维评分。BLEU/ROUGE 仅用于和实验五保持口径
可比，不承担主结论。

## 2. 实验设置

### 2.1 数据、模型与实验矩阵

| 项目              | 设置                                                                              |
| ----------------- | --------------------------------------------------------------------------------- |
| AI 分类样本       | 50 个 PR：37 merged、13 non-merged，沿用实验五固定样本                            |
| AI 生成样本       | 72 个有顶层 inline comment 的 PR，沿用实验五固定样本                              |
| 人类对照          | 实验五 `matched_human_control`，仅在 L4×旧最优与 L4×P5 运行                       |
| 执行模型          | `deepseek-v4-flash`，不训练、不微调、不更换模型                                   |
| 生成裁判          | `deepseek-v4-pro`，四维 1--5 分；hallucination 越低越好                           |
| 旧 Prompt 基线 P* | 分类 P1（Zero-shot）；生成 P3（Chain-of-Thought）                                 |
| 新 Prompt         | P5 Self-Reflection；P6 三轮交互式审查                                             |
| 条件数            | 每个任务 10 格：上下文阶梯 5 格、Prompt 消融 3 格、归因 1 格、对照 2 格（去重后） |

P* 在实验开始前由实验五 AI 预测结果选取：分类 P1 的跨上下文平均 accuracy 最高（0.760），
生成 P3 的平均 BLEU 最高（1.125）。生成侧的选取指标本身并不理想，但它复现了实验五的既定口径，
避免依据实验六结果事后挑选基线。

### 2.2 上下文构建样例

上下文采用确定性、单调递增的 L0--L4 阶梯。每级保留低级内容，并增加一种新的工程证据：

| 级别 | 新增内容                                  | 典型文本块标记                  | AI 分类样本平均字符数 |
| ---- | ----------------------------------------- | ------------------------------- | --------------------: |
| L0   | Python diff                               | 文件路径与 patch                |                 5,916 |
| L1   | PR 标题/正文、全部 commit message         | `## PR 描述`、`## Commit 信息`  |                 8,164 |
| L2   | 修改前后完整函数/类，必要时回退到文件截断 | `## 改前/改后完整函数 (L2)`     |                30,019 |
| L3   | 关联 Issue 正文、历史 review/comment      | `## 关联 Issue 与历史审查 (L3)` |                32,464 |
| L4   | 符号词法检索、同目录文件头、仓库约定      | `## 仓库级检索上下文 (L4)`      |                35,205 |

例如，L3 会把 `Fixes #123` 解析出的 Issue 标题和正文与历史行内审查拼入上下文，并排除作为生成
真值的首条顶层 comment，防止标签泄漏；L4 再加入被改符号在仓库其他文件中的出现片段、同目录相关
文件头部及贡献规范摘要。GitHub API 结果全部落盘缓存，截断顺序固定，保证可复现。

### 2.3 Prompt 设计样例

P5 的关键不是笼统要求“仔细思考”，而是改变决策顺序：

```text
阶段 1：以挑剔 maintainer 视角列出所有 blocking 缺陷；
        若确无缺陷，必须明确写“未发现 blocking 缺陷”。
阶段 2：仅当不存在 blocking 缺陷时判 MERGE；
        存在任一 blocking 缺陷时倾向 REJECT。
输出：blocking_defects + reasoning + decision 的结构化 JSON。
```

P6 将分类拆成“描述改动 -> 专门追问遗漏风险 -> 最终判定”三轮，将生成拆成“意见初稿 ->
自我批评 -> 修订意见”三轮。它比 P5 提供更多反思机会，但也需要三次模型调用。

## 3. 总体结果

![实验六结果总览](results/figures/fig1_result_overview.png)

**图 1 | 输入优化主要改变错误分布，而非全面提高总体性能。** a，L4 下三种 Prompt 的逐类
recall，误差线为 bootstrap 95% CI（分类每格 $n=50$）；b，生成四维裁判分数的等权复合分，
hallucination 按 $6-x$ 反向后纳入（生成每格 $n=72$）。

最重要的总体结果是：**实验六成功改变了模型的审查姿态，但没有发现全面占优的输入方案。**
旧 P1 在 L4 上仍保持 merge recall=1.000，却只能识别 2/13 个 non-merge PR；P5 将
non-merge recall 提高至 0.615，但 merge recall 降至 0.595。生成侧三种 Prompt 的复合质量分
分别为 3.454、3.465、3.462，95% CI 高度重叠，不能宣称 P5 或 P6 全面提高了意见质量。

L0 到 L2 的分类 balanced accuracy 从 0.538 增至 0.583，non-merge recall 从 0.077 增至
0.167，说明完整函数/文件有助于发现少量局部 diff 看不到的问题。但 L3 和 L4 没有继续提高：L3 与
L4 的混淆矩阵完全相同，均为 TN=2、FP=11、FN=0、TP=37。所有区间均覆盖或接近 0.5，样本规模
不足以证明这些点估计差异稳定存在。因此，本实验支持的不是“Repository 上下文必然更好”，而是
**相关上下文可能有用，但无筛选地增加上下文会稀释关键信号**。

| L4 条件                | Balanced accuracy | Non-merge recall | Merge recall |  Accuracy | Parse error | 平均时延/s |
| ---------------------- | ----------------: | ---------------: | -----------: | --------: | ----------: | ---------: |
| P1 旧最优              |             0.577 |            0.154 |    **1.000** | **0.780** |        0.00 |   **3.49** |
| **P5 Self-Reflection** |         **0.605** |        **0.615** |        0.595 |     0.600 |    **0.00** |      15.89 |
| P6 多轮                |             0.593 |        **0.615** |        0.571 |     0.583 |        0.04 |      42.83 |

![Prompt权衡](results/figures/fig3_prompt_tradeoff.png)

**图 3 | Prompt 选择会移动决策工作点，没有方案在所有目标上占优。** a，分类的 non-merge 与
merge recall；b，生成四维质量，其中 non-hallucination 为 $6-\text{hallucination}$。误差线为
95% bootstrap CI。

P5 达到了设计中的直接判据：non-merge recall 相对 P1 提高 0.462，即从识别 2 个 non-merge
增加到 8 个。然而，这不是“免费”的性能提升。P5 同时把 37 个 merged PR 中的 15 个从正确改为
错误拒绝，merge recall 下降 0.405。balanced accuracy 仅提高 0.028，且置信区间高度重叠。

![分类错误转移](results/figures/fig4_error_transitions.png)

**图 4 | Self-Reflection 校正类别偏向，但不保证逐个决策更正确。** a，P1 到 P5 的配对错误迁移；
b，L4 下两种 Prompt 的混淆结构。50 个 PR 逐一配对，无抽样替换。

逐样本配对给出了更清晰的机制证据：在 13 个 non-merge PR 中，P5 保持正确 2 个、新修复 6 个、
仍错 5 个；在 37 个 merged PR 中，P5 保持正确 22 个，却新增 15 个错误。P5 实际上把模型从
“几乎总合并”推向了更保守的工作点。对于安全审查或合并前分流，漏掉高风险 PR 的成本可能更高，
这种变化有业务价值；对于自动合并决策，15 个新增误拒则不可忽略。**Prompt 的优劣必须与错误成本
共同定义，不能只看单个 F1。**

### 4.3 Prompt 与上下文分别解决什么

归因格进一步区分了两者的作用：

| 条件  | Balanced accuracy | Non-merge recall | Merge recall | 解释                                            |
| ----- | ----------------: | ---------------: | -----------: | ----------------------------------------------- |
| L4×P1 |             0.577 |            0.154 |        1.000 | 深上下文保留高 merge recall，但仍明显偏向 merge |
| L0×P5 |             0.436 |            0.385 |        0.486 | 只有批判性姿态时，证据不足导致过度拒绝          |
| L4×P5 |         **0.605** |        **0.615** |        0.595 | 上下文帮助 P5 将“找茬”约束在更有依据的风险上    |

从 L4×P1 到 L0×P5 不能视为单一变量比较，但三格共同说明：Prompt 主要控制**审查姿态和类别偏向**，
上下文主要提供**判断缺陷是否真实的证据**。二者需要联合使用；只有 P5 而缺少上下文时，balanced
accuracy 反而低于随机基准。

## 5. Review Comment Generation 结果

### 5.1 上下文阶梯

| 条件      | Relevance | Actionable |   Correct | Hallucination↓ |      BLEU |   ROUGE-L | 平均执行时延/s |
| --------- | --------: | ---------: | --------: | -------------: | --------: | --------: | -------------: |
| L0×P3     |     1.528 |      3.736 |     3.917 |          1.208 |     0.997 | **0.057** |          11.08 |
| L1×P3     |     1.653 |      3.958 |     3.958 |          1.319 |     0.746 |     0.048 |          14.46 |
| L2×P3     |     1.623 |      3.725 |     3.768 |          1.667 |     1.754 |     0.045 |          16.57 |
| **L3×P3** | **1.800** |  **4.229** | **4.214** |      **1.157** |     1.618 |     0.054 |          13.64 |
| L4×P3     |     1.563 |      4.085 |     3.789 |          1.620 | **2.259** |     0.056 |          15.75 |

生成任务最优的工程上下文是 L3，而不是 L4。L3 同时取得最高 relevance、actionable、correct 和
最低 hallucination；这与其加入 Issue 需求和历史审查意见相符：它们直接描述“代码应该解决什么”
及维护者关注点。L4 的仓库词法片段虽然使 BLEU 升至 2.259，却让 relevance 从 1.800 降至 1.563、
hallucination 从 1.157 升至 1.620。旧词面指标与语义质量给出了相反排序，再次证明 BLEU 不应作为
开放式审查意见的主指标。

不过，L3 与 L4 的 relevance 置信区间仍有重叠，故更严谨的结论是：**L3 的四维点估计一致占优，
值得作为后续默认候选；现有 72 个样本尚不足以确认它稳定优于 L4。**

| L4 条件            | Relevance | Actionable |   Correct | Hallucination↓ |  复合质量 | 平均执行时延/s |
| ------------------ | --------: | ---------: | --------: | -------------: | --------: | -------------: |
| P3 旧最优          |     1.563 |  **4.085** |     3.789 |      **1.620** |     3.454 |      **15.75** |

P5 最明显的收益是 relevance：比 P3 高 0.324，说明先列 blocking 缺陷更容易命中参考意见关注的

执行时延达到 P3 的 2.82 倍。三者复合分相差不超过 0.011，不能据此声称任何 Prompt 全面胜出。

### 5.3 自动生成审查意见样例

在 `apache/airflow#63368` 上，P5 生成：

> 在 `__str__` 中，使用 `if detail is not None:` 代替 truthiness 检查，以确保空字典/空列表等
> 非 None 但有意义的 detail 也能被附加。同时，修复缺失 detail 属性的测试，改用正常构造并
> 手动删除 detail 的方式。

人类参考意见同样指出 `detail` 可能缺失，并强调 `is not None` 能保留 `{}` 或 `""` 等有意义的
falsy 值。裁判给出 relevance=5、actionable=5、correct=5、hallucination=1。该例说明 P5 能在
证据充分时生成高度具体且可操作的意见；但 relevance 总体均值只有 1.887，也说明这种成功尚未覆盖
多数样本。

## 6. 与实验五及人类对照的比较

### 6.1 与实验五基线

实验六的 L0×P1 分类结果与实验五 C1×P1 完全一致（accuracy=0.760、merge-F1=0.860），证明
旧基线被正确复现。L1 对应实验五 C4，但当前实现把 PR 描述与 commit 同时纳入，accuracy 比实验五
C4×P1 低 0.020、merge-F1 低 0.010，差异很小且不影响“旧 Prompt 仍接近全判 merge”的结论。

实验五只能观察到“高 merge-F1”；实验六补充逐类指标后发现旧 P1 的 balanced accuracy 仅
0.538--0.577、non-merge recall 仅 0--0.167。实验六最实质性的改进不是旧头条指标上涨，而是：

- 用 balanced accuracy 与混淆矩阵揭示真实判别力；
- 用 P5 将 non-merge recall 从 0.154 提至 0.615，证明 Prompt 能校正偏向；
- 用裁判四维分数发现 L3 优于 L4 的候选趋势，而 BLEU 给出相反排序。

这也解释了为何 P5 的 accuracy 和 merge-F1 看似“退步”：旧指标奖励多数类偏向，P5 主动牺牲一部分
merge recall 换取 non-merge 识别能力。

## 7. 质量与成本权衡

![质量成本权衡](results/figures/fig6_quality_cost.png)

**图 6 | 多轮 Prompt 显著增加时延，却没有形成清晰质量收益。** 横轴为记录到的平均执行时延，

P6 分类时延为 42.83 s，是 P1 的 12.27 倍、P5 的 2.70 倍；生成时延为 44.42 s，是 P3 的

在实验七的交互式 VSCode 插件中，P6 不适合作为默认同步路径；P5 更适合高风险 PR 的深度审查模式，
旧 Prompt 则适合低延迟预览。需注意，当前 latency 混合缓存与非缓存，适合描述本次实验运行成本，
不等同于严格的在线服务基准。

## 8. 对指导书 6.9 思考题的回答

### Q1. 为什么 Repository 级上下文能够提升代码审查性能？

Repository 上下文理论上能提供局部 diff 缺少的符号用法、同目录约定、测试结构和贡献规范，从而判断
修改是否破坏跨文件契约。本实验只部分支持这一机制：完整函数 L2 改善了分类点估计，Issue 与历史
审查 L3 改善了生成四维质量；但 L4 词法检索没有继续提升，反而可能引入无关片段。正确表述应是
**高相关性的仓库上下文有用，而上下文越多并不必然越好**。后续应优先提升检索精度与证据排序。

### Q2. 哪种 Prompt 最适合 AI 生成代码的代码审查？为什么？

没有跨任务唯一最优 Prompt。若目标是发现可能阻断合并的问题，P5 最合适：它把 AI non-merge recall
从 0.154 提至 0.615，且无解析错误。若目标是低延迟并避免误拒 merged PR，旧 P1 更合适。生成任务
中 P5 relevance 最高，P6 correct 最高，P3 actionable 与 non-hallucination 更好；应按目标选择，
而不是把三者压成单一排名。

### Q3. Prompt 优化与上下文增强分别解决了哪些问题？

Prompt 优化改变模型的**推理姿态与错误成本偏好**：P5 强制主动寻找 blocking 缺陷，抑制“默认
merge”。上下文增强解决**证据不足**：完整函数、Issue 和历史审查帮助验证缺陷是否真实。L0×P5 的
balanced accuracy 只有 0.436，而 L4×P5 为 0.605，说明只有批判性姿态而没有充分证据会过度拒绝。

### Q4. 不同 Prompt 组合是否适用于所有代码审查任务？

不适用。分类需要在 non-merge 与 merge recall 之间选择工作点，生成则同时要求相关、可操作、正确
和少幻觉；P5 在两个任务上的作用也不同。人类对照进一步显示，P5 在 AI 分类上改善、在人类分类上
下降。Prompt 必须按任务、代码来源和错误成本配置，不能全局固定。

### Q5. 未来还可以从哪些方面提升 AI 代码审查效果？

1. 用 AST/符号索引、调用关系与测试映射替代 L4 的纯词法片段，减少上下文噪声。
2. 将“代码质量判断”和“历史合并结果预测”拆成两个输出，显式识别重复 PR、需求变更等流程性拒绝。
3. 引入工具调用运行测试、静态分析、类型检查和安全扫描，让 blocking 缺陷有可验证证据。
4. 使用独立厂商裁判与人工双盲抽样校准 LLM-judge，并报告裁判一致性。
5. 在实验七采用分级路由：低延迟初筛 -> P5 风险复核 -> 工具/Agent 异步深审，而非所有 PR 都走 P6。

## 9. 效度威胁与结论边界

- **样本量与类别不平衡**：分类只有 13 个 non-merge，单个样本就改变 recall 约 0.077；多数
  bootstrap 区间重叠，不应把小幅点估计差异视为显著提升。
- **合并标签噪声**：未合并可能由重复 PR、维护策略或需求变化导致，不等价于代码存在缺陷。P5 判断的
  是 blocking defect，而真值是历史 merge outcome，二者存在目标错位。
- **裁判偏差**：pro 裁判与 flash 执行模型虽不是同一模型，但属于同厂模型；四维分数仍需人工校准。
- **单一参考意见**：每个 PR 只使用首条顶层 inline comment，其他同样合理的审查点会被 relevance
  低估。
- **检索实现边界**：L4 是轻量词法检索，不是语义 RAG，也没有完整调用图；结果不能外推为“仓库级
  语义上下文无效”。
- **时延口径**：记录值混合缓存与非缓存，能够比较本次实验条件成本，但不是生产环境冷启动基准。

## 10. 结论

实验六完成了指导书要求的上下文增强、Prompt 优化、两任务推理、性能比较与实验五对照。最可信的
结论不是“更长上下文或更多轮次必然更好”，而是：

1. 实验五的高 merge-F1 确实掩盖了近似全判 merge；逐类指标是必要测量工具。
2. Self-Reflection 能显著改变类别偏向并提高 non-merge 识别，但会带来大量新增误拒；它是工作点
   调整，不是无条件准确率提升。
3. 完整函数及 Issue/历史审查比无筛选的仓库词法片段更稳定；上下文质量比长度更重要。
4. 生成侧 P5 提高相关性，但三种 Prompt 的综合质量几乎相同；BLEU 与裁判排序冲突，验证了实验五
   对旧生成指标失灵的判断。
5. P6 成本最高且无清晰质量优势，不宜作为实验七默认交互路径。后续系统应采用风险分级、精准检索与
   工具验证，把“模型找茬”转化为可验证的工程证据。
