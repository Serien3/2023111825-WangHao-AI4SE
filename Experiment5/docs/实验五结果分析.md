# 实验五结果分析：针对 AI 生成代码的代码审查（泛化性测试）

> 本文基于实验五全部落盘产物（`Experiment5/results/`）做数据驱动分析，对齐指导书 5.9 五道思考题。
> 实验五定位：实验二 / 四模型的**纯下游消费者**——不重训、不设计新 Prompt，只把已有模型搬到
> AI 生成代码上，量化「人类 vs AI」性能差异并归因。**"性能下降"本身即核心发现，不是失败。**

---

## 1. 实验设置与数据

数据全部来自实验一共享数据基座（`Experiment1/results/processed/`）。AI 代码靠 commit 尾注
`co-authored-by: claude/copilot/cursor` + `ai slop`/`ready for maintainer review` 标签识别。

| 集合 | 定义 | 规模 |
|---|---|---|
| AI 分类池 | AI PR 且含 `python & has_patch` 文件 | 253（187 merged / 66 non，合并率 **74%**） |
| AI 生成池 | AI PR 且含 ≥1 条顶层 inline comment | 72 |
| 匹配人类对照 | 从**实验二 held-out test** 按 AI 的 `repo×is_merged` 分布分层抽 | 50 |

三个实验：

| 实验 | 模型 | AI 侧规模 | 人类锚点 |
|---|---|---|---|
| A. Merge Prediction (ML) | 实验二 SVM/RF/XGB/LGBM × {pre, full} | 全 253 | 实验二 test 旧指标 + 匹配对照 |
| B. Merge Prediction (LLM) | 实验四 16 条件（4 上下文 × 4 Prompt） | 抽 50 | 实验四人类 50 + 匹配对照 |
| C. Review Comment Gen (LLM) | 实验四 16 条件 | 全 72 | 实验四人类 50 |

**复用纪律**：TF-IDF 用固定 50 词表（只 transform）、scaler 只 transform、列序对齐实验二
`PRE/FULL_FEATURES`。已验证 TF-IDF 重建 bit-level 复现实验二（max diff 2e-16）。

**效度关键**：匹配对照组只从实验二 held-out test 抽样，与人类锚点同为**样本外**，且按 AI 分布
匹配以控制"仓库/类别分布"混杂因素。（初版曾误从全人类池抽样导致 84% train/val 泄露，已修复。）

**抽样分布**：分类 AI 50 条 = 74% merged / 26% non-merged（与全量 253 一致）；对照 50 条 =
66% / 34%（test 集某些层容量不足，配额被迫再分配，略偏移）。

## 2. 实验结果

### 2.1 实验 A：Merge Prediction (ML)

pre-review 特征，F1（数据源 `metrics/ml_metrics.json` / `human_vs_ai.json`）：

| 模型 | 人类 test | AI | 匹配对照（无泄露） | AI vs 人类 | AI 侧 Precision / Recall |
|---|---|---|---|---|---|
| SVM | 0.810 | **0.833** | 0.800 | +0.023 | 0.751 / **0.936** |
| RF | 0.813 | **0.585** | 0.862 | **−0.228** | 0.804 / 0.460 |
| XGBoost | 0.798 | **0.594** | 0.812 | **−0.205** | 0.821 / 0.465 |
| LightGBM | 0.804 | **0.630** | 0.767 | **−0.175** | 0.802 / 0.519 |

full 特征集下四个模型在 AI 上 F1 = 0.66–0.69，相对人类 test（0.86–0.87）同样下降 0.17–0.20。

**读法（关键）**：三个树模型在 AI 代码上 F1 崩到 0.58–0.63，而在同为样本外的人类对照上仍
0.77–0.86 —— 这是泛化下降的硬证据，且**非分布假象**（对照已控制分布 + 无泄露）。

**SVM 的 0.833 是假象**：它在 AI 上 recall=0.936、precision=0.75，即几乎把所有 PR 都判成
"合并"。AI 池 74% 是正类，这种近似多数类预测器的 F1 天然虚高。树模型相反——recall 崩到 0.46
（过度保守、漏判正类）。**结论：所有 ML 模型对 AI 代码的判别力都下降，只是表现形式不同。**

### 2.2 实验 B：Merge Prediction (LLM)

16 条件 AI 侧 F1 普遍 0.84–0.86，**高于人类 0.76–0.81**（`gap_ai_vs_human` 全为正）。

**这同样是类别偏向假象**：AI 侧 16 个条件里 recall 几乎全是 1.0 或 0.97，precision 稳定 ~0.75
—— LLM 基本把所有 AI PR 都判成"可合并"，几乎从不 reject。在 74% 正类占比下，这种策略 F1 天然
高。真正的判别力（识别 non-merge 的能力）反而弱。人类侧同样偏高，AI 侧更极端。

parse_error_rate 全条件为 0，说明 JSON 结构化输出稳定，解析无损耗。

### 2.3 实验 C：Review Comment Generation (LLM)

全 16×3 条件：BLEU ≈ 0.5–1.5（满分 100），ROUGE-L ≈ 0.04–0.06。AI 与人类互有高低、差异极小。

**核心发现不是"谁高谁低"，而是 BLEU/ROUGE 在此任务几乎无区分度**：审查意见是开放式文本，与
单一参考意见的 n-gram 重叠必然趋近 0。

> ⚠️ 生成任务的 control 直接复用实验四人类 50 条样本，故 `control` 数值 = `human`，其
> `gap_ai_vs_control` 无独立意义，仅 `gap_ai_vs_human` 有效。

### 2.4 上下文敏感度（C1→C4）

指导书假设"AI 代码更依赖完整上下文"（`metrics/context_sensitivity.json`）：

| 任务 / 指标 | AI 的 C4−C1 增益 | 人类的 C4−C1 增益 | 结论 |
|---|---|---|---|
| 分类 F1 | **−0.001**（几乎为 0） | +0.009 | 加上下文对 AI 分类**无帮助**，假设不成立 |
| 生成 ROUGE-L | **+0.0047**（正） | −0.0052（负） | AI 从上下文获益、人类反而不，**微弱支持**假设 |

### 2.5 错误案例归因

抽取的 5 例典型误判全部来自 pandas，全部是 `true_merge=false` 但模型判 `merge`（假阳性，
`cases/error_cases.json`）。查看 diff 发现这些 AI PR 普遍**带 GH# issue 引用、含单元测试、
注释详尽**，表面质量很高。

**归因**："未合并" ≠ "代码差"——这些多为流程性原因（被重复 PR 取代、维护者决策），属标签噪声。
AI 代码风格统一、表面完整度高，反而更容易"骗过"以代码质量为主的判别器。

## 3. 对齐指导书 5.9 思考题

### Q1. AI 生成代码与人类编写代码有哪些本质区别？

- **风格统一、注释规范、表面完整度高**：错误案例里 AI PR 普遍带 issue 引用、测试、详尽注释，
  外观上"很像会合并的好 PR"。
- **局部正确但整体可能错**：diff 层面看无懈可击，但是否真被合并取决于项目级因素。
- **分布不同于训练语料**：ML 模型在人类代码上学到的 diff→合并映射，在 AI 代码上失效（见 Q2）。

### Q2. 为什么已有模型在 AI 生成代码上性能下降？

- **训练分布偏移**：实验二模型在人类代码上训练，AI 代码的 AST/CFG/文本特征分布不同，树模型
  F1 掉 0.17–0.23，且无泄露对照消融证明这不是"仓库/类别分布假象"。
- **局部 diff 信号变弱**：AI 代码局部质量高，diff 层面的"坏味道"信号减少，模型据此判断的依据
  被削弱——这正是指导书"AI 代码更依赖完整上下文"的动机。
- **标签噪声**：AI PR 的"未合并"常源于流程而非质量，进一步稀释了可学习信号。

### Q3. 哪类模型对 AI 生成代码泛化更好？为什么？

以数据为准，需拆穿两个假象后再判断：

- ML 树模型：F1 明显下降（判别力真实下降，表现为漏判正类）。
- ML SVM / LLM：F1 表面不降甚至升，但都是"类别偏向撞上 74% 高合并率"的假象，判别力同样弱。
- **相对而言，LLM 的优势在生成任务的上下文利用**（C1→C4 有正增益，ML 无此能力），且分类
  parse_error=0、无需重训即可适配新分布。**但在本实验的类别不平衡下，无法仅凭 F1 断言"LLM
  泛化更好"**；更公允的说法是：LLM 对上下文更敏感，具备通过实验六改进输入来提升的潜力。

### Q4. AI 生成代码是否需要新的评价指标？为什么？

**需要。** 生成任务 BLEU ≈ 1、ROUGE-L ≈ 0.05，全条件无区分度——表面 n-gram 匹配指标无法衡量
审查意见的"是否指出真实问题、是否可操作"。分类任务也说明：**单看 F1 会被类别偏向误导**，应同时
报告 precision/recall 甚至按类别分层，或引入语义相似度 / 有用性人工评估。

### Q5. 本实验发现了哪些 AI 生成代码特有的审查挑战？

1. **表面完整度高 → 假阳性**：AI 代码风格统一、测试注释齐全，易骗过以代码质量为主的判别器。
2. **局部信号不足 → 依赖全局上下文**：仅凭 diff 难判质量，需要 repo / issue / 调用关系级上下文
   （实验六的直接动机）。
3. **标签语义漂移**："未合并"混入大量流程性噪声，使监督信号可靠性下降。
4. **评价失灵**：现有表面匹配指标无法有效评估 AI 代码审查产出。

---

## 4. 结论与对实验六的启示

- **Merge Prediction**：已有模型对 AI 代码泛化下降（ML 树模型证据最硬）；F1 虚高处均为类别偏向
  假象，需以 precision/recall 佐证。
- **Review Comment Generation**：BLEU/ROUGE 无区分度，指标本身是瓶颈。
- **上下文假设**：分类任务不成立、生成任务微弱成立；LLM 比 ML 更能利用上下文。
- **实验六方向**（本实验直接支撑）：AI 代码审查的瓶颈在**输入（上下文 + Prompt）而非模型本身**。
  应引入 repo / issue / 调用关系级上下文 + Self-Reflection / 多轮 Prompt，重点改善判别力
  （提升 non-merge 识别、抑制类别偏向）与生成意见的实际有用性，而非追逐 BLEU/ROUGE。
